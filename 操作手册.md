## 项目简介

### 这是什么项目?

这是一个 **Claude Code 基础设施参考库**,包含了经过 6 个月实战测试的各种工具和模式。

**重要提醒:** 这不是一个可以直接运行的应用程序,而是一个参考资料库。你需要把这里的组件**复制到你自己的项目**中使用。

### 为什么要用这个项目?

如果你在使用 Claude Code 时遇到以下问题:
- ✅ 技能(Skills)不会自动激活,总是要手动调用
- ✅ 大型技能文件太长,超出 Claude 的上下文限制
- ✅ Claude 重置会话后,忘记之前的工作内容
- ✅ 每次都要重复解释项目架构和最佳实践
- ✅ 需要专门的代理(Agents)来处理复杂任务

**那么这个项目就是为你准备的!** 它提供了久经考验的解决方案。

## 这个项目是做什么的

### 核心功能

1. **自动激活的技能系统**
   - 当你编辑相关文件或提问时,相关技能会自动被建议
   - 不再需要记住要使用哪个技能

2. **模块化技能设计**
   - 遵循 500 行规则,避免超出上下文限制
   - 主文件 + 资源文件的分层结构

3. **专用代理**
   - 10 个专门的代理处理复杂任务
   - 代码审查、重构规划、文档生成等

4. **开发文档系统**
   - 即使 Claude 重置会话,也能快速恢复工作状态
   - 三文件结构保存所有关键信息

5. **生产级示例**
   - 所有模式都来自真实项目
   - 基于 6 个微服务、5 万行代码的实战经验

## 核心概念解释

### 1. 技能 (Skills) 是什么?

**通俗解释:** 技能就像是 Claude 的"知识模块"或"指导手册"。

**举例说明:**
- **backend-dev-guidelines** 技能 = 后端开发的最佳实践手册
- **frontend-dev-guidelines** 技能 = 前端开发的最佳实践手册
- **skill-developer** 技能 = 如何创建新技能的指南

**使用场景:**
```
你: "我要创建一个新的 API 路由"
Claude (加载 backend-dev-guidelines 技能后):
  "根据项目最佳实践,我会:
   1. 创建路由文件
   2. 创建控制器(使用 BaseController 模式)
   3. 创建服务层
   4. 添加 Sentry 错误追踪
   5. 使用 Zod 进行输入验证"
```

### 2. 钩子 (Hooks) 是什么?

**通俗解释:** 钩子是在特定时刻自动运行的脚本,就像"自动化助手"。

**类型:**
- **UserPromptSubmit** - 你每次发送消息时运行
- **PostToolUse** - Claude 使用工具后运行
- **Stop** - 你点击停止按钮时运行

**核心钩子功能:**
- **skill-activation-prompt** - 分析你的问题,自动建议相关技能
- **post-tool-use-tracker** - 追踪你修改了哪些文件

### 3. 代理 (Agents) 是什么?

**通俗解释:** 代理是独立运行的 Claude 实例,专门处理复杂任务。

**类比:**
- 普通 Claude = 多面手助手
- 代理 = 专家顾问(代码审查专家、重构专家、文档专家等)

**可用的代理:**
- **code-architecture-reviewer** - 审查代码架构
- **refactor-planner** - 规划重构策略
- **documentation-architect** - 生成文档
- **frontend-error-fixer** - 修复前端错误
- **web-research-specialist** - 在线研究技术问题
- 还有 5 个其他专用代理...

### 4. 开发文档 (Dev Docs) 是什么?

**通俗解释:** 一套标准化的文档结构,保存项目状态和进度。

**三文件结构:**
1. **计划文件 (plan.md)** - 整体战略规划
2. **上下文文件 (context.md)** - 当前进度、关键决策、重要文件
3. **任务文件 (tasks.md)** - 待办事项清单

**使用场景:**
- Claude 重置会话? 读取这三个文件,立即恢复工作状态!
- 跨天工作? 不会忘记昨天做了什么!

## 详细使用教程

### 前提条件

**你需要:**
1. 一个现有的项目(任何编程语言都可以)
2. 已安装 Claude Code
3. 项目中有 `.claude` 目录(如果没有,创建一个)

**检查你的项目结构:**
```bash
your-project/
├── .claude/              # 如果没有,请创建这个目录
│   ├── skills/          # 将在后面创建
│   ├── hooks/           # 将在后面创建
│   └── settings.json    # 可能已经存在
├── src/                 # 你的源代码
└── ...
```

---

### 第一步:设置技能自动激活系统

#### 1.1 复制核心钩子文件

**需要复制的文件:**
- `skill-activation-prompt.sh`
- `skill-activation-prompt.ts`
- `post-tool-use-tracker.sh`

**操作步骤:**

```bash
# 1. 创建 hooks 目录(如果不存在)
mkdir -p your-project/.claude/hooks

# 2. 复制技能激活钩子
cp claude-code-infrastructure-showcase/.claude/hooks/skill-activation-prompt.sh \
   your-project/.claude/hooks/

cp claude-code-infrastructure-showcase/.claude/hooks/skill-activation-prompt.ts \
   your-project/.claude/hooks/

# 3. 复制文件追踪钩子
cp claude-code-infrastructure-showcase/.claude/hooks/post-tool-use-tracker.sh \
   your-project/.claude/hooks/

# 4. 赋予执行权限
chmod +x your-project/.claude/hooks/*.sh
```

#### 1.2 安装依赖

```bash
# 如果 hooks 目录有 package.json
cd your-project/.claude/hooks
npm install
```

#### 1.3 配置 settings.json

**如果你还没有 settings.json:**

```bash
# 创建基础配置
cat > your-project/.claude/settings.json << 'EOF'
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/skill-activation-prompt.sh"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|MultiEdit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/post-tool-use-tracker.sh"
          }
        ]
      }
    ]
  }
}
EOF
```

**如果你已经有 settings.json:**

打开你的 `settings.json`,在 `"hooks"` 部分添加:

```json
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/skill-activation-prompt.sh"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|MultiEdit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/post-tool-use-tracker.sh"
          }
        ]
      }
    ]
  }
}
```

#### 1.4 验证安装

```bash
# 检查钩子文件权限(应该显示 -rwxr-xr-x)
ls -la your-project/.claude/hooks/*.sh

# 验证 settings.json 格式正确
cat your-project/.claude/settings.json | jq .
```

✅ **完成!** 技能自动激活系统已就绪。

### 第二步:添加你的第一个技能

#### 2.1 选择合适的技能

**根据你的技术栈选择:**

| 如果你使用... | 选择这个技能 | 需要定制 |
|--------------|-------------|---------|
| Node.js + Express + Prisma | backend-dev-guidelines | ⚠️ 是 |
| React + MUI v7 | frontend-dev-guidelines | ⚠️ 是 |
| 其他后端框架 | 可以改编 backend-dev-guidelines | ⚠️⚠️ 是 |
| 其他前端框架 | 可以改编 frontend-dev-guidelines | ⚠️⚠️ 是 |
| 任何技术栈 | skill-developer (学习创建技能) | ✅ 否 |

**技术栈不匹配?**
- 你可以使用示例技能作为模板,改编成适合你的技术栈
- 或者使用 skill-developer 技能学习如何创建自己的技能

#### 2.2 复制技能目录

**示例:添加 backend-dev-guidelines**

```bash
# 1. 创建 skills 目录
mkdir -p your-project/.claude/skills

# 2. 复制技能
cp -r claude-code-infrastructure-showcase/.claude/skills/backend-dev-guidelines \
      your-project/.claude/skills/

# 3. 验证复制成功
ls -la your-project/.claude/skills/backend-dev-guidelines
```

#### 2.3 配置 skill-rules.json

这是**最关键的步骤** - 告诉系统什么时候激活技能。

**创建 skill-rules.json:**

```bash
# 如果文件不存在,复制模板
cp claude-code-infrastructure-showcase/.claude/skills/skill-rules.json \
   your-project/.claude/skills/
```

**关键:修改路径模式匹配你的项目结构**

打开 `your-project/.claude/skills/skill-rules.json`,找到你添加的技能,修改 `pathPatterns`:

**示例 1: 单体应用**
```json
{
  "backend-dev-guidelines": {
    "fileTriggers": {
      "pathPatterns": [
        "src/**/*.ts",           // 你的后端代码在 src/ 目录
        "api/**/*.ts"            // 或者在 api/ 目录
      ]
    }
  }
}
```

**示例 2: Monorepo(多仓库)**
```json
{
  "backend-dev-guidelines": {
    "fileTriggers": {
      "pathPatterns": [
        "packages/api/src/**/*.ts",      // 第一个后端服务
        "packages/server/src/**/*.ts",   // 第二个后端服务
        "apps/backend/**/*.ts"           // 第三个后端应用
      ]
    }
  }
}
```

**示例 3: 不确定结构,使用通用模式**
```json
{
  "backend-dev-guidelines": {
    "fileTriggers": {
      "pathPatterns": [
        "**/*.ts",          // 所有 TypeScript 文件
        "src/**/*.ts",      // src 目录下的 TS 文件
        "backend/**/*.ts"   // backend 目录下的 TS 文件
      ]
    }
  }
}
```

**验证配置:**
```bash
# 验证 JSON 格式正确
cat your-project/.claude/skills/skill-rules.json | jq .
```

#### 2.4 测试技能激活

**测试方法 1: 编辑相关文件**
1. 在你的后端目录编辑一个 `.ts` 文件
2. Claude 应该会自动建议使用 backend-dev-guidelines 技能

**测试方法 2: 询问相关问题**
```
你: "我要创建一个新的 API 控制器"
Claude: "我建议使用 backend-dev-guidelines 技能..."
```

**如果技能没有激活,请检查:**
- `pathPatterns` 是否匹配你的文件路径?
- hooks 是否正确安装?
- settings.json 配置是否正确?

## 常见问题解答

### Q1: 技能不会自动激活怎么办?

**检查清单:**

### Q4: settings.json 里的 Stop hooks 是什么?我需要吗?

**答案:** Stop hooks 是在你点击"停止"按钮时运行的脚本。

**示例中的 Stop hooks:**
- `tsc-check.sh` - 运行 TypeScript 编译检查
- `trigger-build-resolver.sh` - 编译失败时自动启动修复

**你需要吗?**

**大多数情况: 不需要!**
- Stop hooks 是为复杂的 monorepo 设计的
- 需要大量定制才能适配你的项目
- 如果配置错误会阻塞 Stop 事件

**建议:**
1. 只使用 UserPromptSubmit 和 PostToolUse 钩子
2. 跳过 Stop hooks
3. 需要时手动运行 `tsc --noEmit`

### Q5: skill-rules.json 里的配置项都是什么意思?

**详细解释:**

```json
{
  "skill-name": {
    // 技能类型
    "type": "domain",     // domain = 领域知识, guardrail = 护栏(强制)

    // 强制级别
    "enforcement": "suggest",  // suggest = 建议, block = 阻止(必须使用)

    // 优先级
    "priority": "high",   // high/medium/low - 多个技能匹配时的优先级

    // 提示词触发器
    "promptTriggers": {
      // 关键词:用户消息中包含这些词时触发
      "keywords": ["controller", "route", "API", "endpoint"],

      // 意图模式:正则表达式匹配用户意图
      "intentPatterns": [
        "create.*controller",
        "add.*route",
        "build.*API"
      ]
    },

    // 文件触发器
    "fileTriggers": {
      // 路径模式:编辑这些路径的文件时触发
      "pathPatterns": [
        "src/api/**/*.ts",
        "backend/**/*.ts"
      ],

      // 内容模式:文件包含这些内容时触发
      "contentPatterns": [
        "import.*express",
        "class.*Controller"
      ]
    }
  }
}
```

**常用配置示例:**

**后端技能(建议模式):**
```json
{
  "backend-dev-guidelines": {
    "type": "domain",
    "enforcement": "suggest",
    "priority": "high",
    "promptTriggers": {
      "keywords": ["API", "route", "controller", "service", "backend"]
    },
    "fileTriggers": {
      "pathPatterns": ["src/api/**/*.ts", "backend/**/*.ts"]
    }
  }
}
```

**前端技能(护栏模式 - 防止使用旧版 MUI):**
```json
{
  "frontend-dev-guidelines": {
    "type": "guardrail",
    "enforcement": "block",
    "priority": "high",
    "promptTriggers": {
      "keywords": ["component", "React", "MUI", "UI"]
    },
    "fileTriggers": {
      "pathPatterns": ["src/**/*.tsx"]
    }
  }
}
```

### Q6: 代理和技能有什么区别?我应该用哪个?

**技能 vs 代理对比:**

| 特性 | 技能 (Skills) | 代理 (Agents) |
|------|---------------|---------------|
| **作用方式** | 内联指导 | 独立运行 |
| **何时使用** | 开发过程中 | 完成特定任务 |
| **工作方式** | 提供最佳实践和模式 | 自主执行复杂任务 |
| **用户控制** | 高 - 你保持控制 | 低 - 代理自主工作 |
| **适用场景** | 持续开发 | 一次性任务 |

**使用时机:**

**使用技能当:**
- ✅ 你在写代码,需要指导
- ✅ 想确保遵循最佳实践
- ✅ 需要代码示例和模式
- ✅ 持续开发工作

**使用代理当:**
- ✅ 需要全面的代码审查
- ✅ 规划大型重构
- ✅ 生成复杂文档
- ✅ 调试复杂问题
- ✅ 有明确的任务目标

**两者可以配合:**
```
场景: 开发新功能

1. 开发时使用技能
   你: "创建新的用户服务"
   Claude + backend-dev-guidelines: [提供指导和模式]

2. 完成后使用代理
   你: "用 code-architecture-reviewer 代理审查我的代码"
   代理: [独立审查,提供报告]
```

### Q8: 开发文档(Dev Docs)真的有必要吗?

**取决于你的工作方式:**

**需要开发文档,如果:**
- ✅ 项目工作会跨多天
- ✅ 任务复杂,有很多步骤
- ✅ 经常遇到 Claude 重置会话
- ✅ 需要跟踪大量决策和进度
- ✅ 团队协作,需要文档化

**不需要,如果:**
- ❌ 都是简单的小改动
- ❌ 单个会话就能完成
- ❌ 不需要跨会话延续工作

**实际效果:**

**没有开发文档:**
```
第一天工作 3 小时
[会话重置]
第二天: "呃...我昨天做了什么来着?"
花 30 分钟回忆和重新了解代码...
```

**有开发文档:**
```
第一天工作 3 小时
结束时: /dev-docs-update (2 分钟)
[会话重置]
第二天: "读取开发文档并继续"
Claude 1 分钟内了解所有上下文,直接继续工作
```

**节省时间:** 每次重置节省 20-30 分钟
